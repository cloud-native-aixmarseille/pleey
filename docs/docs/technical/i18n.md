# ЁЯМН Internationalization (i18n) Guide

This guide explains how internationalization is implemented in QuizMaster and how to use it.

## ЁЯУЪ Overview

QuizMaster supports multiple languages through internationalization (i18n):
- **Frontend**: Uses `react-i18next` and `i18next`
- **Backend**: Uses `nestjs-i18n`
- **Languages**: English (en) and French (fr)
- **Default Language**: French (fr) in tests, auto-detected from browser in production

## ЁЯЪи Mandatory Rules

- **No hardcoded user-facing strings**: every visible text must use a translation key that exists in *all* supported locales (currently `en` and `fr`).
- **Error messages come from enums**: define an error enum per domain (`AuthErrorCode`, `QuizErrorCode`, etc.) and always resolve the displayed message through i18n using that enum's translation key.
- **Frontend + Backend alignment**: when you add or change an enum value, update the matching translation entry for both locales in `application/frontend/src/i18n/locales/` and `application/backend/src/i18n/`.
- **Pull requests fail review without translations**: missing locale entries or hardcoded strings must be fixed before merging.

## ЁЯОп Frontend i18n

### Configuration

i18n is configured in `application/frontend/src/i18n/config.ts`:
- Auto-detects browser language
- Falls back to English if language not supported
- Stores language preference in `localStorage`

### Translation Files

Translation files are located in `application/frontend/src/i18n/locales/`:
- `en.json` - English translations
- `fr.json` - French translations

### Using Translations in Components

```tsx
import { useTranslation } from 'react-i18next';

export default function MyComponent() {
  const { t } = useTranslation();

  return (
    <div>
      <h1>{t('home.title')}</h1>
      <p>{t('home.subtitle')}</p>
    </div>
  );
}
```

### Language Switcher

The `LanguageSwitcher` component is available in all pages:
- Located at `application/frontend/src/shared/components/LanguageSwitcher.tsx`
- Appears in top-right corner
- Saves preference to localStorage

To use it:
```tsx
import { LanguageSwitcher } from '../../../shared/components';

<LanguageSwitcher />
```

### Adding New Translations

1. Add the key to both `en.json` and `fr.json`:

```json
// en.json
{
  "mySection": {
    "myKey": "My English text"
  }
}

// fr.json
{
  "mySection": {
    "myKey": "Mon texte fran├зais"
  }
}
```

### Error Messaging

- Declare user-facing errors via enums stored alongside their domain logic (for example, `application/frontend/src/domains/auth/errors.ts`).
- Each enum value must correspond to a translation key (e.g., `auth.errors.invalidCredentials`).
- When displaying an error, resolve the translation with `t(enumKey)` rather than embedding text or enum names directly.
- Tests must assert against translated outputs, never raw enum values.

2. Use in component:
```tsx
{t('mySection.myKey')}
```

## ЁЯФз Backend i18n

### Configuration

i18n is configured in `application/backend/src/app.module.ts`:
- Loads translations from `application/backend/src/i18n/`
- Supports query parameter `?lang=en` or `?lang=fr`
- Uses `Accept-Language` header as fallback
- Default fallback: English

### Translation Files

Translation files are organized by domain in `application/backend/src/i18n/`:
```
application/backend/src/i18n/
тФЬтФАтФА en/
тФВ   тФЬтФАтФА auth.json
тФВ   тФЬтФАтФА game.json
тФВ   тФФтФАтФА quiz.json
тФФтФАтФА fr/
    тФЬтФАтФА auth.json
    тФЬтФАтФА game.json
    тФФтФАтФА quiz.json
```

### Using Translations in Backend

```typescript
import { I18nService } from 'nestjs-i18n';

@Injectable()
export class MyService {
  constructor(private readonly i18n: I18nService) {}

  async doSomething() {
    const message = await this.i18n.translate('auth.errors.invalidCredentials');
    // Use message
  }
}
```

### Adding New Backend Translations

1. Add to appropriate JSON file:

```json
// application/backend/src/i18n/en/auth.json
{
  "errors": {
    "myNewError": "My error message"
  }
}

// application/backend/src/i18n/fr/auth.json
{
  "errors": {
    "myNewError": "Mon message d'erreur"
  }
}
```

### Error Handling & Translation

- Create domain-specific enums for all errors surfaced to API consumers (e.g., `AuthErrorCode.invalidCredentials`).
- When throwing or returning errors, attach both the enum value and the translated message via `nestjs-i18n`.
- Keep translation keys synchronized with the enumsтАФadding a new enum value without translations blocks the change.
- Integration and unit tests must verify the enum-to-translation mapping for both `en` and `fr` locales.

2. Use in code:
```typescript
const message = await this.i18n.translate('auth.errors.myNewError');
```

## ЁЯзк Testing with i18n

### Frontend Tests

Tests use French as the default language (configured in `application/frontend/src/test/setup.js`):

```javascript
import i18n from '../i18n/config';

beforeAll(async () => {
  await i18n.changeLanguage('fr');
});
```

When writing tests, use translated text:
```javascript
// тЬЕ Correct - use translated text
expect(screen.getByText(/Connexion/i)).toBeInTheDocument();

// тЭМ Wrong - hardcoded English
expect(screen.getByText(/Login/i)).toBeInTheDocument();
```

### Backend Tests

Backend tests can specify language using the `lang` parameter:
```typescript
const result = await request(app.getHttpServer())
  .get('/api/endpoint?lang=en')
  .expect(200);
```

## ЁЯОи Design Considerations

The language switcher follows the Cyber Arcade design system:
- Uses `font-display` (Press Start 2P) for uppercase text
- Has `retro-shadow` effect when active
- Positioned fixed in top-right corner
- Backdrop blur with dark background
- Purple primary color for active state

## ЁЯФД Language Detection Flow

### Frontend
1. Check `localStorage` for saved preference
2. Use browser language (`navigator.language`)
3. Fall back to English

### Backend
1. Check `lang` query parameter
2. Check `Accept-Language` header
3. Fall back to English

## ЁЯЪА Best Practices

1. **Always add translations for both languages** when adding new keys
2. **Use descriptive keys** like `auth.errors.invalidCredentials` instead of generic ones
3. **Group related translations** under the same section
4. **Keep translations consistent** between similar UI elements
5. **Test with both languages** to ensure UI doesn't break with longer text
6. **Use placeholders** for dynamic content: `t('game.playerJoined', { name: playerName })`

## ЁЯРЫ Common Issues

### "Missing translation" warnings
- Ensure the key exists in both `en.json` and `fr.json`
- Check for typos in the translation key
- Verify the key path is correct (e.g., `auth.email` not just `email`)

### Language not switching
- Check if localStorage is being cleared
- Verify the language switcher is imported correctly
- Ensure i18n config is imported in `main.jsx`

### Tests failing with translation errors
- Verify test setup initializes i18n (`application/frontend/src/test/setup.js`)
- Use translated text expectations that match the default test language (French)
